Item 1 — Transcript sentence merging

Spec: "combine a full sentence (separated by dot to dot) in a row". Latest constraint: every practice row must contain a full number of English sentences only — exactly one, two, or three sentences, never 1.5 or 2.5 sentences. Target row length is 30-40 words.

Decisions

Sentence-first splitting in src/services/transcript.ts. The transcript pipeline reconstructs raw YouTube fragments into one timed text stream, detects sentence boundaries, then greedily packs only whole sentence units into practice rows. The row packer never splits inside a detected sentence.
  • splitIntoSentenceUnits() scans the reconstructed transcript for ., ?, and !, accepting closing quotes/parens as part of the sentence ending.
  • False positive guards skip decimals (3.14), all-caps acronym-like periods, and common abbreviations such as Dr., Mr., e.g., i.e., U.S., and U.K..
  • packTranscriptLines() groups sentence units into chunks with 1-3 complete sentences. It targets 30-40 words per row: one long sentence remains one row; shorter sentences are combined only when two or three full sentences stay within the target.
  • Timing is preserved by mapping character positions back to source fragment timestamps. A merged row uses the first sentence start and last sentence end.
  • The AI split path is also post-processed through packTranscriptLines() in src/services/preprocess.ts, so both Gemini-generated and local-fallback transcript lines obey the same row-shaping rules.

Tradeoffs / known limits

Transcripts with no punctuation cannot strictly satisfy "full English sentences" locally because there are no reliable sentence boundaries to preserve. In that case the app falls back to mergeByWordCount() at about 45 words to avoid a single giant row. The preferred path is the Gemini /ai/split-transcript preprocessing step, whose prompt now explicitly asks for exactly 1, 2, or 3 complete sentences per segment.

Item 5 — Long-press → Ask AI Tutor

Spec: "long touch on screen and select any vocabulary or sentence, a contextual menu allows me to 'Ask AI Tutor' then open up AI screen above with question like 'Can you explain me <selected_text> in the context of this video?'"

Decisions

Long-press on a transcript line opens a bottom-sheet contextual menu with three actions: Ask AI Tutor, Play from here, Cancel. Tapping Ask AI Tutor opens the AI screen with the line text pre-filled into the prompt: "Can you explain "<selected>" in the context of this video?".

Selection granularity — important deviation

React Native does not expose a cross-platform API for arbitrary text-range selection inside a custom view that lets you intercept a "Look Up" / "Define" menu. The spec phrasing ("any vocabulary or sentence") implies free-form word-range selection. That would require either (a) an iOS-only path using UITextView menu items, or (b) re-rendering the transcript inside a WebView so the browser's native selection works.

Shipped instead: line-level selection. Long-press picks the entire transcript line as the unit. The selected line is shown as a banner at the top of the AI Tutor screen. The user can still ask follow-up questions about a specific word inside the line via free text. If word-level selection becomes important, the cleanest path is a follow-up that adds a "trim selection" sheet between the menu and the AI screen.

Item 7 — Home screen Netflix-style Video Slider

Spec: "bring videos in the library screen into home screen replace for recent video / session sections… change to Netflix style with a slider of 10 videos in a row then next row."

Decisions

Replaced the flat list layout on app/(tabs)/home.tsx with horizontal Netflix-style card sliders chunked using chunkArray(videos, 10). Each video row consists of a title and a horizontal scrolling ScrollView populated by card items. To create a seamless bleed effect, cards extend edge-to-edge of the device by using marginHorizontal: -Spacing.screen on the container combined with matching padding. Each card enforces a fixed width of 240px and a strict aspectRatio: 16/9 for the video thumbnail, overlaid by the video title, channel name, and actions.

  • Videos are client-side chunked into nested matrices: chunks = chunkArray(videos, 10). Each chunk renders as a distinct slider section (e.g., "Library Videos (1-10)", "Library Videos (11-20)").
  • If the user has zero saved videos, the screen falls back to displaying the native empty dashboard view with hints.

Data fetch

  • Home now fans out two requests in parallel on focus: homeService.fetch() (kept for the streak chip) and videoService.list() (the video set). Either can fail independently — a list failure leaves the streak chip working, and vice versa.
  • Removed the recentKnots mapping and the Knot import. The mastery-percent column and per-segment metadata are no longer surfaced on Home; Library has always lacked them too, so behaviour is now consistent across both tabs.

Tradeoffs

Mastery is no longer visible from Home. If we want to bring it back without breaking the new card layout, the cleanest path is a small mastery pill on the right-hand side of each card (matches the existing badge styling). Not in this delivery.

Follow-up decisions — delete + copy actions

Home video rows now expose two inline icon actions: copy URL and delete. Copy uses expo-clipboard and shows a small fading bottom toast instead of a blocking Alert.alert. Delete remains destructive-confirmed, calls videoService.remove(id), and performs an optimistic UI removal.
  • videoService.remove(id) calls DELETE /videos/:id, so deletion is DB-backed rather than local-only.
  • The backend route checks ownership by user_id, deletes related practice_sessions first, then deletes user_videos.
  • A 404 / not-found delete response is treated as already removed from the DB, so the row stays removed instead of rolling back. Other failures restore the previous list and show an error alert.
  • Home video data is fetched from the API on focus and is not persisted in app local storage. The local persisted stores are auth/session, AI settings, preprocessed transcripts, and saved phrases.

Item 10 — Transcript split stabilization

Problem: the current splitter in src/services/transcript.ts treats a detected sentence as atomic — it never splits inside a sentence even when the "sentence" is hundreds of words long. YouTube auto-captions frequently produce sparse punctuation, so the algorithm emits monster chunks. The test runner confirms this:

cd app && npm run test:transcript -- --url https://www.youtube.com/watch?v=RzcD5jQgZeI

→ Chunk #4: 279 words, 1 sentence  (target 30–40)
→ Chunk #10: 166 words, 1 sentence

Goal: every chunk lands in the 20–45 word range without giving up the sentence-boundary protection that keeps clean transcripts clean.

Decision — Hybrid local + synchronous AI fallback

Two-stage pipeline inside src/services/transcript.ts. Stage 1 is deterministic soft-boundary splitting (no network). Stage 2 is a synchronous AI call only for chunks that remain oversized, routed through the existing provider-agnostic Config.aiEndpoint so the app stays decoupled from any specific LLM vendor — the backend picks the provider.

Stage 1 — Local soft-boundary splitting

  • Add splitLongSentence(sentence, times). When a detected sentence exceeds MAX_WORDS_PER_CHUNK (40), split it on soft boundaries in this priority order:
    1. ; and :
    2. , followed by whitespace
    3. Coordinating conjunctions at word boundary: and, but, so, because, or, yet, nor
    4. Common spoken fillers: you know, I mean, like, right, well
  • Apply greedily but only when the left piece is at least MIN_WORDS_PER_CHUNK (30). If no soft boundary yields a valid split, fall back to a hard word-count cut at UNPUNCTUATED_WORDS_PER_CHUNK (45) — delegate to mergeByWordCount rather than duplicating the cut logic.
  • Timestamps survive by reusing the existing buildTimedText / lineFromSpan machinery (transcript.ts:102, 150). Soft splits produce sub-spans of the same times[] array, so each new chunk gets accurate start / duration without recalculation.
  • splitIntoSentenceUnits invokes splitLongSentence on every detected sentence that exceeds MAX_WORDS_PER_CHUNK before returning to packTranscriptLines. packTranscriptLines itself is unchanged.

Stage 2 — AI fallback for residuals

  • After splitTranscriptLocally runs, scan the chunk list. A chunk is sent to AI if it is both > AI_FALLBACK_WORD_CEILING (proposed: 60) and contains ≤ 1 sentence boundary.
  • Synchronous during fetchTranscript — the transcript load awaits the AI call. Acceptable because (a) only the residual chunks are sent, not the whole transcript, (b) most videos won't trigger it at all, (c) it keeps the data flow simple.
  • Request payload (per chunk): 
    POST {Config.aiEndpoint}/v1/resegment
{
  "text": "<chunk text>",
  "targetMinWords": 20,
  "targetMaxWords": 45
}
  • Expected response: 
    { "splits": [
  { "text": "...", "wordOffset": 0 },
  { "text": "...", "wordOffset": 23 }
] }
  • AI text is never trusted for word content — only wordOffset is consumed. The client re-maps each wordOffset to a character index in the original chunk text, then uses the existing times[] slice to assign timestamps. This makes the fallback robust against hallucination.
  • Per-video in-process cache keyed by videoId avoids re-calling within the same session. Persistent caching is out of scope.

Failure handling

AI failure must never break transcript loading. If the endpoint is unreachable, returns a non-2xx, returns an invalid shape, or any wordOffset fails range validation against the original chunk, the Stage 1 chunk is preserved as-is and a console.warn is logged. The user sees the locally-split result, which is already much better than today.

Backend contract

The endpoint expected by the app — the backend team picks the underlying provider (Gemini, Claude, OpenAI, etc.) and prompt:

POST {apiBaseUrl-or-aiEndpoint}/v1/resegment
Content-Type: application/json

{
  "text": "",
  "targetMinWords": 20,
  "targetMaxWords": 45

→ 200 {

    { "text": "", "wordOffset": 0 },
    { "text": "", "wordOffset": 23 },

wordOffset is the zero-based word index in the original text where each sub-chunk starts. The first sub-chunk must have wordOffset: 0. Sub-chunks should each fall in the target word range; the client is responsible for tolerating mild violations.

Item 11 — Session restoration + Sign out + Home delete deploy fix

Problem: two adjacent bugs surfaced while debugging "delete video" on Home.

  1. Delete returned 404 in production. The local Worker code had DELETE /videos/:id but the deployed Worker at soundknot-api.ndkhoa-is.workers.dev did not. The client's "treat 404 as already removed" branch swallowed the failure silently — the row vanished optimistically and reappeared on next Home focus.
  2. Home occasionally rendered an empty video list. Reproduced after Metro Fast Refresh / Reload while sitting on Home. Logs showed GET /videos → "Missing or invalid Authorization header" immediately after a successful DELETE. The bearer token had vanished from the apiClient singleton mid-session.

Root cause — singleton token desync

apiClient in src/services/api.ts holds the bearer token in a private field that lives in JS memory only. The token was set in exactly four call sites (login / register / logout / restoreSession) inside authStore. restoreSession only ran inside app/index.tsx's useEffect — and index.tsx only mounts when the user lands on the / route. Any code path that bypassed / (Fast Refresh while on /(tabs)/home, an OS-evicted JS VM rehydrating directly into a deep route) left apiClient.token = null while AsyncStorage still held a valid session. Every protected request then 401'd with the auth-middleware message.

Fixes shipped

FileChange
app/_layout.tsxCalls useAuthStore.restoreSession() in a useEffect at app boot and gates the entire stack render on authChecked. No child route can mount before the persisted token is re-attached to apiClient.
src/stores/authStore.tsAdded a module-level useAuthStore.subscribe(...) that mirrors session?.access_token into apiClient.setToken/clearToken on every change. The store is now the single source of truth; the singleton can't drift again even if a future code path forgets to call setToken.
app/index.tsxStripped to a single <Redirect>. Auth restoration now belongs to the root layout, so this route doesn't repeat the work or duplicate the splash UI.
app/(tabs)/progress.tsxAdded an Account section at the bottom of the Profile tab showing the user's email and a destructive-confirmed Sign out pill button. Calls useAuthStore.logout() then router.replace('/login').
Deployed WorkerRan npx wrangler deploy from the api/ repo so the DELETE /videos/:id handler is live. The route itself was unchanged.

Tradeoffs / known limits

No token refresh. Supabase access tokens expire after ~1 hour. The fix above guarantees the saved token is always attached, but it does not refresh an expired token mid-session. Long-running app sessions can still 401 once the token expires. The proper follow-up is to detect response.status === 401 in api.ts, attempt one refresh_token exchange via the existing Supabase session, retry the request once, and only then bubble the error.
Home still hides backend errors as an empty list. useFocusEffect in home.tsx swallows both homeService.fetch and videoService.list failures with .catch(() => null / { videos: [] }). A 401/5xx looks identical to "you have 0 videos". Worth a small follow-up: render an explicit "Could not load — tap to retry" state instead.
Delete-error swallowing on 404 stays in place. The Worker is now deployed so this branch should not fire in practice, but if the API ever drifts again the user will see no warning. Acceptable for now; revisit if we add more destructive actions.

  2. Press r in Metro to reload while on Home → videos still load (this was broken before).
  3. Trigger Fast Refresh by editing any file → videos still load after the reload.
  4. Profile tab → bottom → tap Sign out → confirm → land on /login.

Item 13 — AI Tutor Prompt Refactoring & Parity with Playground

Problem: The mobile app returned highly compressed, unformatted bullet points or flat text blocks from gemini-3.1-flash-lite, while the developer playground yielded beautifully structured Markdown analyses with headings, subheadings, and bold vocabulary lists using the same model.

Root cause

The standard tutor PREAMBLE prompt imposed rigid answer shape templates (e.g. limiting summary responses to single-level unformatted bullet lists with "clean bullets", restricting explanation to exactly 2-4 short paragraphs, and dictating strict length limitations). The playground, operating under a minimal system prompt, allowed the model's natural rich-formatting instructions to flourish.

Fixes

Refactored the system PREAMBLE in both the app client app/src/services/aiTutor.ts and the server proxy api/src/lib/aiSystemPrompt.ts to encourage structured, detailed, and high-quality educational Markdown formatting:
  • Replaced rigid shape guidelines with explicit instructions for rich formatting: descriptive headings (### H3), sub-headings, paragraph explanations, and styled bullets for summaries.
  • Instructed the model to use bolding and structure (definitions, concrete usage examples, listening/pronunciation tips) instead of flat paragraphs for vocabulary explanations.
  • Retained vital system rules (the literal [t=MM:SS] timestamp seeker token format, user language mirroring, and priority of explicit user format instructions).
  • Deployed the Hono worker changes via wrangler deploy to soundknot-api.ndkhoa-is.workers.dev so both proxy-mode and direct-mode calls achieve identical high-quality formatted replies.

Item 15 — Keyboard Tap Persistence Configuration

Problem: When the input URL text bar was focused, pressing the execution/submission button required a double-tap (the first tap dismissed the keyboard, and only the second tap actually registered on the button).

Decisions

Added keyboardShouldPersistTaps="handled" to the outermost ScrollView wrapper inside app/(tabs)/home.tsx. This ensures that any press event on interactive buttons (like the URL submit button or chips) is intercepted and handled immediately without the keyboard swallow-dismissing the first touch gesture.

Item 16 — Spotify-style dark theme redesign

Spec: "I love Spotify design. can you copy all 12 Spotify-style cover design and apply into this app". Iterated through many follow-ups: switch app to dark mode, remove old Netflix cards, redesign Home and Library with reusable cover components, add a featured "Today's Pick" card, and add a "Trending Videos" portrait slider.

Decisions

Replaced the warm-paper light theme with a Spotify-inspired dark theme. Built four new presentation components under src/components/spotify/ that render real UserVideo data (no static mock playlists). Home and Library are now thin orchestrators that pick which video components to show.

Theme switch — Spotify dark palette

  • src/constants/theme.ts — useTheme() now returns DarkColors; useIsDark() returns true. The light-only lock from §4.1 of technical-docs.html is gone.
  • src/constants/Colors.ts — DarkColors rewritten to true Spotify values: paper #121212, paper2 #1A1A1A, ink #FFFFFF, accent #1DB954 (Spotify green), positive #1DB954, white-on-transparent hairlines.
  • ColorTokens type widened from typeof LightColors (literal-typed) to a string-based interface so both palettes satisfy the contract. This unlocks future dark/light toggling without further type churn.

The thumbnail URL uses maxresdefault.jpg (1280×720) rather than hqdefault.jpg (480×360) for sharper rendering at the hero card's full-width size.
