Item 1 — Transcript sentence merging

Spec: "combine a full sentence (separated by dot to dot) in a row". Latest constraint: every practice row must contain a full number of English sentences only — exactly one, two, or three sentences, never 1.5 or 2.5 sentences. Target row length is 30-40 words.

Decisions

Sentence-first splitting in src/services/transcript.ts. The transcript pipeline reconstructs raw YouTube fragments into one timed text stream, detects sentence boundaries, then greedily packs only whole sentence units into practice rows. The row packer never splits inside a detected sentence.
  • splitIntoSentenceUnits() scans the reconstructed transcript for ., ?, and !, accepting closing quotes/parens as part of the sentence ending.
  • False positive guards skip decimals (3.14), all-caps acronym-like periods, and common abbreviations such as Dr., Mr., e.g., i.e., U.S., and U.K..
  • packTranscriptLines() groups sentence units into chunks with 1-3 complete sentences. It targets 30-40 words per row: one long sentence remains one row; shorter sentences are combined only when two or three full sentences stay within the target.
  • Timing is preserved by mapping character positions back to source fragment timestamps. A merged row uses the first sentence start and last sentence end.
  • The AI split path is also post-processed through packTranscriptLines() in src/services/preprocess.ts, so both Gemini-generated and local-fallback transcript lines obey the same row-shaping rules.

Tradeoffs / known limits

Transcripts with no punctuation cannot strictly satisfy "full English sentences" locally because there are no reliable sentence boundaries to preserve. In that case the app falls back to mergeByWordCount() at about 45 words to avoid a single giant row. The preferred path is the Gemini /ai/split-transcript preprocessing step, whose prompt now explicitly asks for exactly 1, 2, or 3 complete sentences per segment.

Item 5 — Long-press → Ask AI Tutor

Spec: "long touch on screen and select any vocabulary or sentence, a contextual menu allows me to 'Ask AI Tutor' then open up AI screen above with question like 'Can you explain me <selected_text> in the context of this video?'"

Decisions

Long-press on a transcript line opens a bottom-sheet contextual menu with three actions: Ask AI Tutor, Play from here, Cancel. Tapping Ask AI Tutor opens the AI screen with the line text pre-filled into the prompt: "Can you explain "<selected>" in the context of this video?".

Selection granularity — important deviation

React Native does not expose a cross-platform API for arbitrary text-range selection inside a custom view that lets you intercept a "Look Up" / "Define" menu. The spec phrasing ("any vocabulary or sentence") implies free-form word-range selection. That would require either (a) an iOS-only path using UITextView menu items, or (b) re-rendering the transcript inside a WebView so the browser's native selection works.

Shipped instead: line-level selection. Long-press picks the entire transcript line as the unit. The selected line is shown as a banner at the top of the AI Tutor screen. The user can still ask follow-up questions about a specific word inside the line via free text. If word-level selection becomes important, the cleanest path is a follow-up that adds a "trim selection" sheet between the menu and the AI screen.

Item 6 — AI integration

Architecture

Two-mode service in src/services/aiTutor.ts:
  1. Proxy (default): POST {apiBaseUrl}/ai/chat with { provider, model, messages, context }. The bearer token from authStore is attached automatically by apiClient. The backend holds the API key — this matches the answer you gave.
  2. Direct (fallback): calls Gemini's generativelanguage.googleapis.com/v1beta/models/{model}:generateContent using a key the user pastes in Settings. Useful before the proxy endpoint ships, and as an escape hatch for testing different keys.
Provider type is an enum ('gemini' only today) so adding OpenAI or Anthropic later is a one-file change.

Backend contract — what the server needs to expose

The proxy endpoint expected by the app:

POST /ai/chat
Authorization: Bearer <user session token>
Content-Type: application/json

{
  "provider": "gemini",
  "model": "gemini-3.5-flash",
  "messages": [
    { "role": "user", "content": "what does 'in the saddle' mean?" }
  ],
  "context": {
    "videoTitle": "...",
    "videoChannel": "...",
    "videoId": "...",
    "transcriptWindow": "...",
    "selection": "..."
  }
}

→ 200 { "reply": "..." }
→ non-2xx { "error": "..." }

The server is responsible for: holding the Gemini key, prepending the system prompt, and (for voice) accepting messages[*].audio = { base64, mimeType } and forwarding it as Gemini inlineData. The exact same payload shape works for OpenAI Whisper if you swap providers later.

Voice input

Native: records via expo-av with HIGH_QUALITY preset, reads the resulting file as base64, attaches it to the next user message as audio = { base64, mimeType }. Gemini's inlineData path accepts audio directly and does transcription + answer in one round-trip — no separate STT call. This matches your "use the configured AI provider's STT" answer.
Web: not implemented (per §12.4 of technical_docs.md, the existing dictation mic was already a UI-only sim on web). The button shows an inline error on web; users can still type.

Mic permissions: already declared in app.json for iOS (NSMicrophoneUsageDescription) and Android (RECORD_AUDIO). No app.json changes needed.

Context window

buildTranscriptWindow(lines, currentSeconds, halfWindow=15) picks every transcript line that overlaps [t-15, t+15]. If nothing matches (e.g. before playback starts) it falls back to the first 5 lines. The Listen screen passes this window plus video title/channel into navigation params when opening AI Tutor.

Settings UX

  • Two-segment toggle: Backend proxy vs Direct (Gemini).
  • Model picker shows the current Gemini lineup (gemini-3.5-flash, gemini-2.5-pro, gemini-3.1-pro, gemini-3-flash-preview, gemini-3.1-flash-lite).
  • Key field is masked by default with a show/hide eye; persists on blur.
  • "Test connection" button sends a single "Say 'ok' if you can read this." message and shows the response or error inline.

Routing

  • Two new Stack.Screen entries in app/_layout.tsx: ai-tutor (modal, slide-from-bottom) and ai-settings (slide-from-right).
  • router.push calls cast the new pathnames as any in two places. Reason: Expo Router auto-generates typed routes at .expo/types/router.d.ts only when the dev server runs; tsc --noEmit on a fresh checkout doesn't see the new routes yet. Restarting expo start regenerates the types — at that point the casts can be removed.

Item 7 — Home screen video list

Spec: "bring videos in the library screen into home screen replace for recent video / session sections."

Decisions

Replaced the Knot-based "Recent videos / sessions" list on app/(tabs)/home.tsx with the same video card pattern used on Library: 64×64 thumbnail, title, channel. Same data source (videoService.list()), same row layout, same tap-to-Listen navigation. The home screen now shows the user's full saved video set under a "Your videos" header, directly below the URL paste row.

Data fetch

  • Home now fans out two requests in parallel on focus: homeService.fetch() (kept for the streak chip) and videoService.list() (the video set). Either can fail independently — a list failure leaves the streak chip working, and vice versa.
  • Removed the recentKnots mapping and the Knot import. The mastery-percent column and per-segment metadata are no longer surfaced on Home; Library has always lacked them too, so behaviour is now consistent across both tabs.

Tradeoffs

Mastery is no longer visible from Home. If we want to bring it back without breaking the new card layout, the cleanest path is a small mastery pill on the right-hand side of each card (matches the existing badge styling). Not in this delivery.

Follow-up decisions — delete + copy actions

Home video rows now expose two inline icon actions: copy URL and delete. Copy uses expo-clipboard and shows a small fading bottom toast instead of a blocking Alert.alert. Delete remains destructive-confirmed, calls videoService.remove(id), and performs an optimistic UI removal.
  • videoService.remove(id) calls DELETE /videos/:id, so deletion is DB-backed rather than local-only.
  • The backend route checks ownership by user_id, deletes related practice_sessions first, then deletes user_videos.
  • A 404 / not-found delete response is treated as already removed from the DB, so the row stays removed instead of rolling back. Other failures restore the previous list and show an error alert.
  • Home video data is fetched from the API on focus and is not persisted in app local storage. The local persisted stores are auth/session, AI settings, preprocessed transcripts, and saved phrases.

Item 8 — Bookmarks per transcript line + Library Phrase/Vocabulary tabs

Spec: "add bookmark icon under timestamp along the transcript for each sentence. when user touch on bookmark, save this sentence into library screen. In library screen, there are two vertical tabs : Phrase and Vocabulary… clean up all videos listing in the Library screen."

Decisions

New persistent store at src/stores/savedPhrasesStore.ts (Zustand + AsyncStorage, mirrors the aiSettingsStore shape). Each saved entry is { id, text, kind: 'phrase' | 'vocabulary', source?: 'video' | 'ai', videoId, videoTitle, videoChannel, start, createdAt }, persisted under soundknot_saved_phrases. Lookups use a composite key of videoId@startMs so toggling on the transcript is O(n) but stable across re-renders.
  • Listen screen: the timestamp column became a small "gutter" — timestamp on top, bookmark TouchableOpacity directly under it. Tapping the bookmark calls addPhrase / removeByLine and toggles between bookmark-outline and the filled bookmark Ionicon (accent orange when saved). The inner Touchable swallows the press, so the row's onPress={() => seekTo(line.start)} still fires for the rest of the row.
  • Default kind = phrase. Transcript bookmarks save as phrases. AI Tutor generated keyword lists save as vocabulary; generated grammar/phrase lists save as phrases.

Library rewrite

Old app/(tabs)/library.tsx (saved videos list + search) replaced wholesale. New screen has a two-segment pill tab bar (Phrases / Vocabulary) with per-tab counts, search, time filters (All, Yesterday, Last 7 days), and compact cards for saved entries. Cards show source badges, parsed title/explanation/example lines, source metadata, and inline actions. Tapping any card opens a detail bottom sheet with full content and contextual actions.
  • AI Tutor generated saves are marked source: 'ai'. Library treats them as AI-curated, disables video navigation, and hides the Open video action. Older AI saves from before the source field are detected by their large synthetic start value.
  • Transcript-saved items stay source-linked. Their cards and detail sheet can open /listen at the saved timestamp.
  • Generated AI list items are parsed by the "title - explanation Transcript/Example: ..." pattern. The title renders bold, explanation renders on the next line, and example/transcript renders on its own italic line.
  • The previous Move to Phrase/Vocabulary action was removed from both cards and the detail menu to reduce cognitive load.
  • Copy/delete feedback uses a small fading bottom toast, not a blocking confirmation alert. Delete remains available from cards and detail sheet.

Tradeoffs

Saved videos are no longer reachable from Library. Per the spec ("clean up all videos listing in the Library screen"), the video list and search field are gone. Home already shows the full video set (Item 7), so this is intentional — but if the app grows past the Home list capacity, we'll want a separate "Videos" tab rather than reintroducing the list here.
No automatic lemmatisation for vocabulary. The Vocabulary tab stores whole sentences just like Phrase — kind is just a tag. If you want true single-word vocab, the cleanest follow-up is a "trim selection" sheet on the Library row that lets the user pick a span before promoting to Vocabulary.
