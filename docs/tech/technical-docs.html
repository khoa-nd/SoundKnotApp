Part 1 · Backend / API

Architecture

[Expo App (React Native)] │ │ HTTPS + Bearer JWT ▼ [Cloudflare Worker (Hono)] │ │ supabase-js (service_role) ▼ [Supabase Auth + PostgreSQL]
  • The app never talks to Supabase directly — the CF Worker is the sole API proxy
  • Supabase Auth handles email/password authentication
  • The CF Worker validates JWTs via supabase.auth.getUser(token)
  • AsyncStorage on the app persists JWT for session restoration on launch

Authentication

End-to-end, who issues the token, who carries it, who validates it, and how the app keeps it attached across reloads. Details for each piece are linked inline.

Identity provider — Supabase Auth

  • Email + password only. No social login, no email verification (registration uses supabase.auth.admin.createUser({ email_confirm: true }) server-side, so the user is usable immediately).
  • Supabase issues a standard pair on login or registration: a short-lived access token (JWT, ~1 hour TTL) and a refresh token.
  • The app and the CF Worker both treat Supabase as the source of truth for identity — neither stores password hashes itself. User records live in Supabase's auth.users table; application-level rows in profiles / user_progress are auto-created via DB trigger (on_auth_user_created).

App side — where the token lives

  • Persistent storage: AsyncStorage under soundknot_session (the full session object: access_token + refresh_token + expires_at) and soundknot_user (the cached user record).
  • In-memory store: useAuthStore (Zustand) — owns the session object and the isAuthenticated flag. See §7.1 authStore.
  • HTTP client: apiClient singleton in src/services/api.ts — holds the bearer token in memory and attaches it as Authorization: Bearer <token> to every request.
  • Sync between them: the store registers a module-level useAuthStore.subscribe(...) that mirrors session?.access_token into apiClient.setToken/clearToken on every change. The store is the single source of truth; the singleton can't drift.
  • Boot-time restore: app/_layout.tsx calls authStore.restoreSession() once on mount and gates the entire route stack on the result — so the token is always re-attached before any child screen makes a request, including after Fast Refresh or an OS-evicted JS VM. See §7.2 Auth + State Lifecycle.

Backend side — how requests are validated

  • Every route group except /auth/register and /auth/login is wrapped by authMiddleware (api/src/middleware/auth.ts).
  • The middleware extracts Bearer <token>, calls supabase.auth.getUser(token) to validate against Supabase, and on success sets c.set("userId", user.id) for handlers to use. Missing/malformed header → 401 "Missing or invalid Authorization header". Invalid/expired token → 401 "Invalid or expired token".
  • Once userId is on the context, every handler scopes its queries by it (e.g. .eq("user_id", userId)) so cross-user access is impossible at the application layer.
  • The CF Worker talks to Supabase using the service role key (SUPABASE_SERVICE_ROLE_KEY) which bypasses RLS. The middleware is therefore the only enforcement point at the API layer. RLS policies remain enabled at the DB as defense-in-depth, but the runtime path never relies on them.
  • Endpoint-level details: see §API · Auth Middleware and the per-route Auth column in the endpoint tables.

Failure modes & current gaps

  • Token expiry mid-session: Supabase access tokens last ~1 hour. The app does not currently exchange the refresh token when a request returns 401 — the user has to sign out and back in. The Home tab's .catch(() => ({ videos: [] })) swallows this 401 and renders an empty list with no warning. Tracked as a follow-up.
  • 404 on DELETE is treated as "already removed" client-side — a routing/deployment drift between client and server can silently masquerade as a successful delete. Acceptable for now since the route is deployed.
  • No multi-device session invalidation: logging out on device A does not invalidate the token on device B. Standard Supabase behavior; out of scope.

MethodPathRequest BodyResponse
POST/sessions{ video_id, segment?, pass?, mastery?, accuracy?, listened_seconds?, tz? }{ session: PracticeSession }

MethodPathResponse
GET/home{ progress, todaySessions, recentKnots, videos }

AI Tutor (requires auth)

MethodPathRequest BodyResponse
POST/ai/chat{ provider: "gemini", model, messages: [{ role, content, audio? }], context? }{ reply } on 200, { error } otherwise
  • Strict provider allowlist: only "gemini" is accepted today (returns 400 for anything else)
  • messages[].audio carries voice questions: { base64, mimeType }. Gemini transcribes and answers in one round-trip via inlineData parts
  • context (optional) carries videoTitle, videoChannel, videoId, transcriptWindow, selection — used to build the system prompt server-side; empty/null fields are skipped cleanly
  • Per-user sliding-window rate limit (default 20/min) returns 429 with a Retry-After header. In-memory and per-isolate (best-effort)
  • Audio cap (default 2 MB base64) returns 413
  • Upstream error mapping: Gemini 429 → 429; other Gemini 4xx/5xx, network errors, empty replies, or safety blocks → 502
  • Logs { user_id, provider, model, messages_length, audio_count, latency_ms, upstream_status, outcome } per request — never logs message content, audio, or transcript
  • Spec: specs/api-handoff-ai-chat.md

1. Project Overview

Sound Knot is an English listening practice app built with Expo (React Native) and runs on iOS, Android, and the web. The core philosophy: "Work hard now, listen smart later." Users accumulate deliberate listening hours by engaging with authentic expert content (podcasts, lectures, conversations), using active recall and dictation to build deep comprehension.

The V2 design was prototyped in HTML/JSX at design/SoundKnotAppV2Design/ and translated into this React Native codebase. The design system is documented in design/SoundKnotAppDesignV2.pdf.

5.2 YouTube Player — Platform-Split

src/components/youtube/YoutubePlayerView.{web,native,tsx}

The player is split across three files so Metro picks the right implementation per platform:

FilePlatformImplementation
YoutubePlayerView.native.tsxiOS / AndroidWraps react-native-youtube-iframe (which uses react-native-webview internally)
YoutubePlayerView.web.tsxWebNative HTML <iframe> to youtube.com/embed/{id} + YT IFrame postMessage API
YoutubePlayerView.tsxTypeScriptRe-exports from .native so tsc --noEmit resolves types

Shared interface:

interface YoutubePlayerHandle {
  getCurrentTime: () => Promise<number>;
  getDuration: () => Promise<number>;
  seekTo: (seconds: number) => Promise<void>;
}
interface YoutubePlayerViewProps {
  videoId: string;
  width: number;
  height: number;
  play: boolean;
  onReady: () => void;
  onStateChange: (state: 'playing' | 'paused' | 'ended') => void;
  onError: (error: string) => void;
}

Web implementation details:

  • Embeds https://www.youtube.com/embed/{id}?enablejsapi=1&origin={origin}&...
  • Listens to window.message events, parses YT JSON payloads (onReady, onStateChange, onError, infoDelivery)
  • Caches currentTime / duration from infoDelivery events; getCurrentTime() posts a listening event then waits 100ms and returns the cached value
  • Sends playVideo / pauseVideo / seekTo commands via postMessage

Why platform split: a runtime if (Platform.OS !== 'web') require(...) guard does not work — Metro statically resolves all require() calls at bundle time. Using the .web.tsx / .native.tsx file convention lets Metro pick the correct file per platform during bundling.

10. Key Algorithms

10.1 Knot Rendering (src/components/ui/Knot.tsx)

Parametric trefoil knot rendered via react-native-svg:

  1. Parametric generation (360 steps): 
    x = cx + (sin(t) + 2·sin(2t)) × scale
y = cy + (cos(t) - 2·cos(2t)) × scale
z = sin(3t)
scale = (size/2) × 0.28 × (1 - mastery × 0.12)
  2. Over/under weave: split curve at z > 0.3, draw paper-colored gap strokes, redraw over-strands.
  3. Played portion: amber stroke-dasharray proportional to progress.
  4. Pass rings: concentric dashed circles (1 per completed pass).
  5. Tick marks: 12 perimeter ticks; central dot sized by mastery.

10.5 Transcript Sentence Chunking (src/services/transcript.ts)

YouTube returns transcripts as ~2-3 second fragments. SoundKnot treats transcript rows as recall-practice chunks, not subtitle cues. The splitter prefers complete sentence boundaries when captions are clean, but now includes a local soft-boundary stabilizer for sparse-punctuation auto-captions. Target chunk size is 20-45 words; the stabilizer prevents single malformed "sentences" from becoming hundreds of words.

Stage A — Rebuild timed text

  1. buildTimedText(fragments) normalizes fragment text and concatenates it into one transcript string.
  2. For every character in the reconstructed string, it stores an approximate timestamp derived from the source fragment's start and duration.
  3. This lets later sentence chunks preserve useful start/duration even when a sentence crosses several raw YouTube fragments.

Stage B — Detect sentence units, then stabilize oversized units

  1. splitIntoSentenceUnits() scans the rebuilt text for sentence terminators: ., ?, and !, including closing quotes/parens after the terminator.
  2. sentenceBoundaryEnd() rejects common false positives:
    • Decimals, e.g. 3.14
    • All-caps acronym-like periods
    • Common abbreviations: Mr., Mrs., Dr., Prof., e.g., i.e., U.S., U.K., etc.
  3. lineFromSpan() converts each sentence span back into a TranscriptLine with text, start, and duration.
  4. If a detected sentence exceeds MAX_WORDS_PER_CHUNK, splitLongSentenceSpan() splits the same timed span on soft boundaries before returning it to the packer.

Soft-boundary priority

  1. Punctuation: ;, :, and ,
  2. Coordinating conjunctions at word boundaries: and, but, so, because, or, yet, nor
  3. Common spoken fillers: you know, I mean, like, right, well
  4. Hard fallback: cut at HARD_SPLIT_WORDS = 45 when no soft boundary gives a valid split.

Each soft split is timestamp-preserving: it reuses the original times[] array and emits sub-spans through lineFromSpan(). AI text is not involved in Stage B.

Stage C — Pack only whole sentences

  • packTranscriptLines() greedily groups sentence units while preserving sentence boundaries.
  • Constants:
    • MAX_SENTENCES_PER_CHUNK = 3
    • MIN_WORDS_PER_CHUNK = 20
    • MAX_WORDS_PER_CHUNK = 45
    • AI_FALLBACK_WORD_CEILING = 60
  • If sentences are short, it combines up to 3 consecutive full sentence units when the combined row stays within the 20-45 word target.
  • If a detected sentence is oversized, it has already been split into soft-boundary units before this pack step.
  • Every merged row uses start from the first sentence and duration through the last sentence.

Stage D — optional AI fallback for residuals

  • fetchTranscript(videoId) calls applyAiFallback(localLines) after local splitting.
  • A chunk qualifies only if it is both above AI_FALLBACK_WORD_CEILING = 60 words and has <= 1 sentence ending.
  • The fallback calls aiService.resegmentChunk(text, { targetMinWords: 20, targetMaxWords: 45 }), which posts to ${Config.aiEndpoint}/v1/resegment.
  • Returned AI text is ignored. Only wordOffset is consumed, then remapped onto the original chunk text/timestamps by remapAiWordOffsets(). Invalid offsets or network failures preserve the local chunk and log console.warn.
  • The transcript review runner defaults to local-only. Use --with-ai to exercise the full fetchTranscript() path.

Fallback — no punctuation

  • If the transcript has no sentence punctuation at all, local code cannot strictly know English sentence boundaries. In that case mergeByWordCount() flushes every UNPUNCTUATED_WORDS_PER_CHUNK = 45 words.
  • For messy ASR, the local soft-boundary stabilizer runs first. AI fallback is reserved for residual oversized chunks, not the full transcript.

10.11 AI Transcript Splitting Endpoint (api/src/routes/ai.ts)

POST /ai/split-transcript (authenticated, rate-limited via existing checkRateLimit).

Request:

{
  "videoId": "abc123",
  "transcript": [
    { "text": "hello world", "start": 0.0, "duration": 2.1 }
  ]
}

Behavior:

  1. Validates body and per-line shape (text non-empty string, start/duration finite numbers).
  2. Builds a timestamped transcript block [start-end] text per line.
  3. Calls Gemini (gemini-1.5-flash, temperature: 0.1, maxOutputTokens: 8192) with a JSON-only system prompt instructing it to emit exactly 1, 2, or 3 complete English sentences per segment. The target is 30-40 words: keep one long sentence as one segment, and combine only consecutive full sentences when the combined segment fits the target.
  4. Strips optional ```json fences from the reply and parses as JSON.
  5. Validates each segment (text string, finite start, finite duration), normalizes whitespace, clamps start ≥ 0 and duration ≥ 0.5.

Response: { "lines": [{ text, start, duration }, …] }.

Failure modes:

  • 400 — invalid request body
  • 429 — rate limit exceeded
  • 500 — GEMINI_API_KEY not configured
  • 502 — Gemini upstream error, blocked response, empty output, or unparseable JSON

The endpoint logs a structured ai_split_transcript event with user_id, video_id, fragment count, segment count, latency, and outcome.

Appendix A · How Supabase Auth works

SoundKnot's entire identity story sits on top of Supabase Auth. This appendix is a concept primer — what Supabase Auth actually is, what each piece does, and where each piece lands in our codebase. If you've never used Supabase before, read this first; the per-file detail is linked back to the rest of the doc.

A.1 What is Supabase?

Supabase is an open-source backend-as-a-service that bundles four things behind a single dashboard and a single project URL:

ComponentWhat it isUsed by SoundKnot?
PostgreSQLA full Postgres database, accessible over the wireYes — all our tables live here
Auth (GoTrue)A user-management service: signup, login, JWTs, refresh tokens, password reset, social providersYes — email + password only
StorageS3-style file storageNo
Realtime / Edge FunctionsWebSocket subscriptions + serverless functionsNo (we use Cloudflare Workers instead)

Each Supabase project gets its own URL like https://<project-ref>.supabase.co. Everything below — Auth API, Postgres, REST shortcuts — lives under that URL.

A.2 The Auth service (GoTrue)

The "Auth" piece is a separate service called GoTrue that Supabase runs for you. It exposes HTTP endpoints like /auth/v1/signup, /auth/v1/token, /auth/v1/user, and it owns its own table in your database called auth.users. You never touch auth.users directly — GoTrue is the only thing that writes to it.

Three things to remember about GoTrue:

  1. It stores users in auth.users (id, email, encrypted password, metadata, timestamps). Your application tables live in the public schema and reference auth.users(id) via foreign keys.
  2. It issues JWTs signed with a project-wide secret (the JWT_SECRET) — anyone who knows that secret can verify a token without a network call.
  3. It handles password hashing (bcrypt), email verification flows, magic links, OAuth, MFA — none of which you have to implement.

A.3 The two keys you'll see — anon vs service role

Every Supabase project ships with two API keys. Knowing the difference is essential because they grant very different power.

KeyRole claimWhat it can doWhere it's safe to use
anon public key"anon"Whatever your Row Level Security policies allow for an anonymous user. Subject to RLS.Front-end clients (mobile apps, browser code)
service_role secret key"service_role"Full read/write on every table. Bypasses RLS. Can call admin APIs (create users, delete users, etc.).Server-side only. Never ship to a client.

SoundKnot uses only the service role key — and only inside the Cloudflare Worker (SUPABASE_SERVICE_ROLE_KEY wrangler secret). The app never sees it. The app never talks to Supabase directly at all — it goes through our CF Worker, which holds the privileged key.

This is the simpler of the two common patterns. The alternative — letting the mobile app call Supabase directly with the anon key + RLS — is what most Supabase docs show. We chose the Worker-as-proxy pattern instead so we have one server-side place to add rate limits, custom logic (YouTube oEmbed lookup, AI calls), and observability without exposing any provider keys to the client.

A.4 The JWT pair — access token and refresh token

On successful login or registration, GoTrue returns a session object that contains:

{
  "access_token":  "<JWT>",       // short-lived, ~1 hour
  "refresh_token": "<opaque>",     // long-lived, swappable
  "expires_at":    1748102400,     // unix seconds
  "token_type":    "bearer",
  "user":          { ... }
}

The access token

  • A signed JWT with three parts: header, payload, signature.
  • Payload includes sub (= the user's UUID), email, role ("authenticated" for normal users), exp, iat.
  • Signed with the project's JWT_SECRET. Any server that knows the secret can validate the token offline (no Supabase round-trip needed).
  • ~1 hour lifetime. After that, all requests carrying it will be rejected.

The refresh token

  • An opaque random string (not a JWT).
  • You exchange it at POST /auth/v1/token?grant_type=refresh_token for a brand-new access token (and usually a new refresh token).
  • Long-lived. Stored on the client and persisted across app restarts.
  • Refresh tokens themselves get invalidated on explicit sign-out or on password change.

How the two work together (in any Supabase app)

Client GoTrue ────── ────── login(email, pwd) ────► issues access + refresh ◄──── { access, refresh, expires_at } (every API call) Authorization: Bearer <access> ────► your server validates JWT (access expires after ~1h) refresh_token=<refresh> ────► issues new access (+ new refresh) ◄──── { access', refresh', expires_at' }

SoundKnot today implements only the first two steps. The third — automatic refresh when an access token expires — is not yet wired up. That's why an open app left running for >1 hour starts to 401; the user has to sign out and back in. See the "Failure modes" subsection of §Authentication.

A.5 Row Level Security (RLS)

RLS is a Postgres feature, not a Supabase invention — but Supabase makes it central because it lets you safely expose your database to untrusted clients (the anon-key pattern). With RLS enabled on a table, every query is filtered by SQL policies you define. Example from specs/migration.sql:

CREATE POLICY "Users can view own videos"
  ON public.user_videos FOR SELECT
  USING (auth.uid() = user_id);

auth.uid() is a Supabase helper that pulls the sub claim out of the JWT carried on the request. The policy above means: a SELECT from user_videos using the anon key will only see rows where user_id matches the caller's UUID. No way for one user to read another's videos, no application code required.

In SoundKnot: RLS is enabled on every table and we have per-user policies — but because the Worker uses the service role key, those policies are bypassed at runtime. We rely on the Worker's authMiddleware (which sets userId on the request context) and .eq("user_id", userId) filters in every query to enforce isolation. The RLS policies stay in place as defense-in-depth in case someone ever introduces a direct-client path or the service role key leaks.

A.6 Database triggers — auto-creating application rows

Because Supabase Auth owns auth.users, you typically want a row in your own profiles table to appear automatically whenever a user signs up. Standard pattern: a Postgres trigger that fires on INSERT INTO auth.users.

SoundKnot uses exactly this pattern (see §DB Schema · Triggers):

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

The trigger inserts one row into profiles (display name from signup metadata or email prefix) and one row into user_progress (zeros for all stats). Without it, every signup path would need to remember to bootstrap those rows by hand.

A.7 How a server validates a JWT — two options

When your backend receives Authorization: Bearer <jwt>, it can verify the token two ways:

MethodNetwork call?Used in SoundKnot
supabase.auth.getUser(token)Yes — Supabase round-tripYes (current implementation)
Verify JWT signature locally with JWT_SECRETNoNot used

The local-verify path is faster (no network hop) and stateless, but the server-call path is safer against token-revocation edge cases (e.g. user signs out on another device — local verify wouldn't notice until the token expires). SoundKnot chose getUser() for simplicity and correctness; latency is fine for our traffic shape.
